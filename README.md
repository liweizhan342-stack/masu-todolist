# Masu TodoList

一個以 **低維護成本、容易理解、容易部署** 為核心原則打造的 TodoList 小專案，
目前已從單純清單升級為更像 **daily dashboard** 的版本。

它仍然刻意避免一開始就走大型框架、後端服務或複雜基礎設施，
但已把 UI、互動與資訊呈現拉到更實用、更舒服的層級。

---

## 這個版本的方向

這次升級不是把專案硬做大，而是做三件事：

1. **更快看出現在該做什麼**
2. **更低阻力地修改與收斂任務**
3. **讓專案更容易持續維護**

所以你會看到它開始有：

- hero summary / 今日焦點 / 完成率
- Today / Overdue / High Priority 視角
- 焦點模式（只看值得先處理的任務）
- 側邊推薦任務摘要
- 雙擊直接編輯任務文字
- localStorage 舊資料鍵值遷移
- 從單一 HTML 拆成 `index.html + styles.css + app.js`

---

## 功能

目前版本包含：

- 新增任務
- 勾選完成 / 取消完成
- 刪除任務
- 雙擊直接編輯任務
- 篩選：全部 / 進行中 / 已完成 / 今天 / 逾期 / 高優先
- 關鍵字搜尋（即時過濾）
- 焦點模式（只看高優先、今天到期、逾期）
- 任務優先級：低 / 中 / 高
- 截止日期
- 統計摘要：總數 / 未完成 / 已完成 / 完成率 / 今日焦點
- JSON 匯出 / 匯入
- 拖拽排序（Drag & Drop Reorder）
- PWA / 離線支援（manifest + service worker）
- 使用 `localStorage` 儲存資料（重新整理後不消失）
- 舊版 localStorage key 自動遷移到新版儲存鍵

## 技術選擇

目前仍刻意維持精簡：

- `HTML`
- `CSS`
- `JavaScript`
- `localStorage`
- `SortableJS`
- `Web Manifest`
- `Service Worker`

### 為什麼不急著上框架？

因為這個專案的目標不是展示技術棧，而是：

- 看得懂
- 改得動
- 部署快
- 不會把未來的自己搞死

如果之後需求真的成長到：

- 多頁面
- 複雜狀態同步
- 雲端資料同步
- 多人協作

那時候再評估更重的架構，會比現在就先把複雜度塞進來更合理。

## 這次升級重點

### 1. UI 從「工具頁」升級成「工作台」
不是只換皮，而是讓使用者打開頁面時先看到：

- 現在有多少任務
- 今天有多少需要先處理
- 完成率是多少
- 最值得先做的是哪幾件

### 2. 把「找焦點」的成本降下來
新增：

- Today filter
- Overdue filter
- High Priority filter
- Focus-only mode

這樣不用每次都靠眼睛掃完整份清單。

### 3. 降低修正阻力
原本任務如果打錯字、想改 wording，通常要刪掉重建。
現在可以：

- 雙擊文字直接改
- 或按編輯按鈕修改

### 4. 維護性提升
原本功能越來越多後，全部塞在 `index.html` 會開始變難維護。
所以這次把：

- 結構放回 `index.html`
- 樣式獨立到 `styles.css`
- 邏輯拆到 `app.js`

這讓後續再加功能時比較不容易失控。

## 部署方式

這個專案仍然非常適合部署到 **GitHub Pages**。

原因：

- 不需要伺服器
- 不需要資料庫
- 不需要後端
- 成本低
- 純靜態即可運作

如果未來要升級，也可考慮：

- Netlify
- Vercel

但目前 GitHub Pages 已經足夠。

---

## 後續可考慮的升級路線

建議仍維持「低風險、高可見價值優先」：

1. 任務標籤（tag）
2. 任務備註 / 子任務
3. 自訂排序方式（依優先級、依到期日）
4. 夜間 / 白天主題切換
5. 可選的雲端同步
6. 行動裝置手勢優化

---

## 核心原則

這個專案不是在追求做成最大、最炫、最像 SaaS 的 Todo app。
它真正的目標是：

- **真的能用**
- **真的能持續改**
- **真的能快速部署**
- **真的適合 AI 協作，但不犧牲人類可維護性**

---

## 總結

Masu TodoList 現在已經不是只有「記錄待辦」，
而是開始往「幫你判斷現在該推哪件事」的方向前進。

而且它仍保留了最重要的東西：

> **簡單、清楚、低維護。**
