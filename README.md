# Masu TodoList

一個以 **低維護成本、容易理解、容易部署** 為核心原則打造的 TodoList 小專案。

這個專案刻意避免一開始就走大型框架、後端服務或複雜基礎設施，而是先完成一個：

- 真的能用
- 真的看得懂
- 真的能持續改
- 真的能快速部署

它同時也是一個很適合 **AI 協作開發** 的起點：功能明確、範圍可控、修改成本低。

---

## 功能

目前版本包含：

- 新增任務
- 勾選完成 / 取消完成
- 刪除任務
- 篩選：全部 / 進行中 / 已完成
- 關鍵字搜尋（即時過濾）
- 任務優先級：低 / 中 / 高
- 截止日期
- JSON 匯出 / 匯入
- 使用 `localStorage` 儲存資料（重新整理後不消失）

---

## 開發思路

這個專案的策略不是「先做最強」，而是「先做最穩」。

核心原則如下：

1. **先用靜態前端完成主要價值**  
   不先依賴後端與資料庫，降低出錯面與維護成本。

2. **先做 MVP，再逐步升級**  
   每次只增加少量功能，例如優先級、截止日、搜尋，而不是一次膨脹成複雜系統。

3. **維護性優先於炫技**  
   不為了看起來厲害而塞入過多抽象層、工具鏈或框架。

4. **讓人類能接手**  
   就算這個專案有 AI 參與，也要確保未來人類讀得懂、改得動、接得住。

---

## 技術選擇

目前採用的技術非常精簡：

- `HTML`
- `CSS`
- `JavaScript`
- `localStorage`

這樣做的理由：

### 1. 學習成本低
不需要先理解 React、Vue、打包器、路由、狀態管理等概念，就能直接看到成果。

### 2. 維護成本低
整體檔案少、邏輯集中、部署簡單，適合初期快速迭代。

### 3. 部署成本低
純靜態專案可直接部署到 GitHub Pages、Netlify、Vercel 等平台。

### 4. AI 協作效果好
AI 容易生成、修改、解釋這種專案；人類也比較容易驗收與接手。

## 專案結構

```text
todolist/
├── index.html
└── README.md
```

目前刻意維持極簡結構，避免過早切分檔案造成理解成本上升。

等功能繼續增加到一定程度後，再考慮拆分為：

- `index.html`
- `styles.css`
- `app.js`
- `manifest.json`
- `sw.js`

這樣的拆分才有意義。

## 目前實作重點

### 1. 任務資料模型
目前每個任務至少包含：

- 文字內容
- 完成狀態
- 優先級
- 截止日期
- 建立時間
- 唯一 ID

這樣的資料模型雖然簡單，但足以支撐後續的大多數擴充。

### 2. 篩選與搜尋分離
專案中同時存在兩種過濾邏輯：

- 狀態篩選（全部 / 進行中 / 已完成）
- 關鍵字搜尋

這兩者不直接修改原始資料，而是只影響渲染結果。這樣可讓資料來源保持乾淨，後續維護更容易。

### 3. 本地持久化
透過 `localStorage` 儲存任務資料，避免使用者重新整理後資料消失。

### 4. JSON 匯出 / 匯入
這提供了很實用的資料備份能力，也讓跨裝置轉移資料有初步基礎。

## 部署方式

這個專案很適合部署到 **GitHub Pages**。

原因：

- 不需要伺服器
- 不需要資料庫
- 不需要額外後端
- 成本低
- 可直接透過網址跨裝置存取頁面

如果未來要再往上升級，也可以考慮：

- Netlify
- Vercel

但目前 GitHub Pages 已足夠。

---

## AI 參與方式

這個專案可以坦白標示為：

> **由 AI 協助開發，但以可維護性為優先。**

這裡最重要的不是「全部都由 AI 寫」，而是：

- AI 協助加速開發
- 人類保留理解與判斷
- 每次升級都維持可讀性

如果 AI 只會一次噴出大框架，最後通常會把維護者搞死。這個專案的目標正好相反：

> **用 AI 做出成果，但不把未來的自己炸死。**

## 後續升級路線

目前已記錄的下一步候選方向：

1. 拖拽排序（Drag & Drop Reorder）
2. PWA / 離線支援（manifest + service worker）

建議順序仍然是：

- 先做低風險、高可見價值的功能
- 再做需要引入額外概念或外部庫的功能

---

## 維護建議

如果未來要持續升級，建議遵守這些原則：

- 一次只加一個明確功能
- 每加一個功能，就重新確認是否仍然易懂
- 不要過早導入大型框架
- 先確保需求真的存在，再增加複雜度
- README 跟功能一起更新，不要讓文件落後

---

## 總結

這個專案目前不是在追求最完整的 Todo 系統，
而是在建立一個良好的基礎：

- 能開發
- 能部署
- 能維護
- 能逐步長大

這樣未來不管是你自己改、讓 AI 幫你改、或和其他工具協作，都比較不容易失控。
